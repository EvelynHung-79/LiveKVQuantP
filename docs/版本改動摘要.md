0302 → 0306 的改動摘要
Latency 優化（有效的部分）
1. attention_core.py

把手動的 scale → softmax → matmul 換成 F.scaled_dot_product_attention() (SDPA)，自動選最快 backend（Flash Attention）
prefill / decode / prefill-with-history 三路 causal mask 分離，decode 時完全不算 mask
2. model_wrapper.py

decode loop 裡 position_ids 不再每步 torch.tensor([[pos]]) 重新建立，改成預分配後 fill_() 原地更新
3. kv_cache_manager.py

decode buffer 預分配：decode 時 token 插入從 O(n) concat 變成 O(1) buffer write，這是 latency 最大的來源
4. engine.py

latency 量測拆成 prefill / decode / end-to-end 三段
trec、samsum、lcc、repobench-p 改用 completion 模式（不套 chat template）
Memory 為什麼反而變多（+3,830 MB）
有三個新的開銷：

Raw chunk 暫存：prefill 時先 store_raw() 存 FP16，attention 算完後才 finalize_last_chunk() 壓縮。長文本（narrativeqa）的 chunk 多，短暫同時持有 FP16 + 壓縮版
Lazy reconstruction cache：反覆讀 KV 時 cache 住重建後的 tensor，避免重複解壓，但多佔了一份
Decode buffer 預分配：為了 O(1) 寫入，預先配置大 buffer
所以你說的觀察完全正確——0306 的 memory 增加不是 bug，是這些 latency 優化的 trade-off 代價。到 0310 的方案 C（INT4 packing）才把這些開銷壓回去。
